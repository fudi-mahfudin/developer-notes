# 3Sum

## 1) Ringkasan Soal

**Info soal (isi dulu agar latihan terarah)**

- **Tingkat kesulitan:** Medium
- **Topik utama:** Sorting, two pointers, deduplikasi
- **Inti masalah:** Temukan semua triple unik (urutan indeks) yang berjumlah nol, biasanya tanpa duplikat triple.

---

- Soal: Dari `nums`, kembalikan semua `[a,b,c]` dengan `a+b+c=0`, tanpa duplikat (nilai bisa sama jika indeks berbeda, tergantung soal — LC: triple unik by value).
- Input: `nums: number[]`
- Output: `number[][]`
- Constraints utama: O(n²) target; hindari O(n³) brute dengan dedup yang rapi.
- Pattern utama (Array / HashMap / Two Pointers / DP / Graph / dll): Sort + two pointers

## 2) Jawaban Ideal Singkat (30-60 detik)

> Saya sort dulu agar bisa mengunci elemen pertama `i` dan menyelesaikan 2-sum pada sisa dengan two pointers `L,R` untuk target `-nums[i]`. Untuk menghindari triple duplikat, saya skip `i` jika `nums[i] === nums[i-1]`, dan setelah menemukan satu triple, geser `L` dan `R` sambil skip nilai sama. Kompleksitas O(n²) waktu, O(1) atau O(log n) untuk sort tergantung bahasa/ruang sort — dominan O(n²). Edge: banyak nol.

Struktur cepat:
- Observasi inti masalah: Fix satu elemen mengurangi ke 2-sum pada array terurut.
- Strategi final yang dipilih: Sort + nested loop `i` + two pointers.
- Kenapa strategi ini paling cocok: Dedup alami setelah sort; hash approach lebih rumit untuk «unique triples».
- Time complexity: O(n²) (setelah O(n log n) sort)
- Space complexity: O(1) ekstra selain output (sort in-place umumnya)
- Edge case utama: semua nol; tidak ada jawaban; jawaban banyak.

## 3) Versi Ultra Singkat (10-20 detik)

> Sort; untuk tiap i, two pointer cari pasangan jumlah = -nums[i]; skip duplikat di i, L, R.

## 4) Pseudocode Ringkas (5-10 baris)

```text
sort(nums)
result = []
for i in 0..n-3:
  if i > 0 and nums[i] == nums[i-1]: continue
  if nums[i] > 0: break
  L = i + 1; R = n - 1
  while L < R:
    s = nums[i] + nums[L] + nums[R]
    if s == 0: append; advance L and R skipping duplicates
    else if s < 0: L++
    else: R--
return result
```

## 5) Implementasi Final (Inti Saja)

```js
function threeSum(nums) {
  nums.sort((a, b) => a - b);
  const res = [];
  const n = nums.length;
  for (let i = 0; i < n - 2; i++) {
    if (i > 0 && nums[i] === nums[i - 1]) continue;
    if (nums[i] > 0) break;
    let L = i + 1, R = n - 1;
    while (L < R) {
      const s = nums[i] + nums[L] + nums[R];
      if (s === 0) {
        res.push([nums[i], nums[L], nums[R]]);
        L++; R--;
        while (L < R && nums[L] === nums[L - 1]) L++;
        while (L < R && nums[R] === nums[R + 1]) R--;
      } else if (s < 0) L++;
      else R--;
    }
  }
  return res;
}
```

## 6) Bukti Correctness (Wajib)

- Invariant / alasan logika benar: Untuk `nums[i]` tetap, pasangan `(L,R)` mencari semua solusi 2-sum terurut pada suffix; skip duplikat tidak menghilangkan triple unik baru.
- Kenapa semua kasus valid tercakup: Masing-masing `i` memeriksa semua kombinasi valid di kanan tanpa tumpang tindih yang tidak perlu karena `L` dan `R` monoton.
- Kenapa tidak ada kasus yang terlewat: Sort menjaga urutan; break saat `nums[i] > 0` aman untuk sum 0 karena semua berikutnya positif.

## 7) Dry Run Singkat

- Kasus normal: `[-1,0,1,2,-1,-4]` → `[[-1,-1,2],[-1,0,1]]` (urutan LC).
- Kasus edge: `[0,0,0]` → `[[0,0,0]]`.
- Hasil: Triple unik sesuai.

## 8) Red Flags (Yang Harus Dihindari)

- O(n³) triple loop tanpa dedup yang baik.
- Set hash untuk triple — bisa O(n²) ruang dan masih perlu pola dedup.
- Lupa skip setelah menemukan triple sehingga duplikat output.

## 9) Follow-up yang Sering Muncul

- Follow-up 1 (optimasi):
  - Jawaban ideal singkat: 3-sum terdekat; k-sum generalisasi (rekursif + two pointer).
- Follow-up 2 (trade-off):
  - Jawaban ideal singkat: Hash one-pass untuk 2-sum tanpai sort — lebih sulit dedup triple.
- Follow-up 3 (variasi soal):
  - Jawaban ideal singkat: Hitung jumlah triple (bukan enumerate) — modifikasi logika tanpa menyimpan semua.

## 10) Trade-off Keputusan

- Opsi A: Sort + two pointers — standar.
- Opsi B: Hash frequency untuk variasi tertentu — jarang lebih sederhana untuk LC 3-sum.
- Kenapa memilih opsi final: Dedup + kompleksitas jelas.
- Risiko dari opsi final: Off-by-one dan skip duplikat salah.
- Mitigasi: Pola skip konsisten (bandingkan dengan `i-1`, `L-1`, `R+1`).
